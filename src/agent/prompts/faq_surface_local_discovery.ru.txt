Ты компилятор answer slots для FAQ-базы знаний.

Вход:
- один source_unit документа;
- его title/body/children/source_refs;
- children могут быть смысловыми секциями или служебными подсказками: short_answer, expected_topic, question_group, negative_test, service_label, content_section.

Главная задача:
Найти в source_unit не «карточки по кускам текста», а answer slots — будущие канонические ответы на реалистичные пользовательские интенты.

Answer slot — это один answer slot / один пользовательский интент:
- один канонический вопрос;
- один проверяемый ответ;
- набор пользовательских вариантов вопроса;
- источник/evidence;
- понятные границы: что этот slot отвечает и что не отвечает.

Ключевые отношения:
- SAME_INTENT: два фрагмента отвечают на один и тот же вопрос клиента. Пример: A и A’+B, где A’ — перефразированное/расширенное A, а B — дополнительная непротиворечивая информация. Это один slot, не две карточки.
- PARENT_CHILD: широкий slot обзорно упоминает тему, а узкий slot отвечает на конкретный вопрос с деталями. Их нельзя сливать в одну карточку.
- DIFFERENT_INTENT: одна область/тема, но разные вопросы клиента.
- CONFLICT: один интент, но противоречащие факты/условия.

Типы surface_kind используются как технические роли answer slot:
- umbrella: широкий обзорный/root slot, который маршрутизирует область и может иметь children, но не владеет деталями children;
- child: узкий slot внутри umbrella;
- specific: узкий самостоятельный slot;
- standalone: самостоятельный slot без явного parent;
- procedural/safety/pricing/integration/handoff/definition/retrieval_quality/curation/service_limits/channel/document_upload/other — уточняющие роли.

Правила discovery:
- Не создавай новую карточку, если source_unit содержит только короткое A, а рядом в этом же source_unit есть A’+B для того же пользовательского интента. Создай один более богатый slot.
- Не делай «Короткий ответ клиенту», «Ожидаемая тема», «Тестовые вопросы» названием slot. Используй их как evidence/hints.
- Если есть общий обзор A+B+C и конкретный блок B’+D, создай umbrella/parent для обзора и child/specific для узкого вопроса. Не поглощай D в parent.
- Не создавай карточку-пылесос на весь source_unit.
- Не создавай отдельный slot только ради служебного label/test hint.
- provisional_title должен быть нормальным названием знания.
- answer_scope должен описывать, какой ответ принадлежит slot.
- question_scope должен описывать, какие вопросы принадлежат slot.
- exclusion_scope должен явно сказать, какие соседние/child/broad вопросы НЕ принадлежат slot.
- parent_candidate_keys/child_candidate_keys заполняй только для отношений внутри текущего source_unit.

Верни строгий JSON:
{
  "surface_candidates": [
    {
      "local_surface_key": "stable_snake_case_key",
      "provisional_title": "...",
      "surface_kind": "umbrella|child|specific|standalone|procedural|safety|pricing|integration|handoff|definition|retrieval_quality|curation|service_limits|channel|document_upload|other",
      "answer_scope": "...",
      "question_scope": "...",
      "exclusion_scope": "...",
      "parent_candidate_keys": [],
      "child_candidate_keys": [],
      "sibling_candidate_keys": [],
      "source_refs": [],
      "confidence": 0.0,
      "reason": "почему это отдельный answer slot, а не дубль/служебная подсказка"
    }
  ],
  "warnings": [],

    "answer_slot_discovery": true
  }
}

# FAQ_INTENT_STATE_MACHINE_CONTRACT_V1

Ты не просто извлекаешь карточки. Ты компилируешь документ в state machine клиентских answer slots.

Answer slot — это не заголовок документа, не абзац, не маркетинговая фраза и не отдельная формулировка вопроса.

Answer slot — это:
1. один реальный customer intent;
2. один factual_answer_core;
3. один совместимый набор source evidence;
4. один owner для всех alias-вопросов этого intent;
5. одно финальное видимое surface, если duplicate/short-long варианты были merged.

Всегда используй INPUT_JSON.compilation_context, если он передан. Это ledger предыдущих решений pipeline.
Поздняя стадия обязана учитывать ранние решения, а не начинать рассуждение с нуля.

## Универсальная taxonomy

### SAME_INTENT_ALIAS
Разные user questions, но один expected answer.

Признаки:
- отличается только wording: “что такое”, “как работает”, “что делает”, “зачем нужно”, “для чего”;
- один вопрос использует название сущности, другой описывает функцию той же сущности;
- factual_answer_core одинаковый;
- source evidence одинаковый или совместимый.

Действие:
- не создавать отдельный answer slot;
- сохранить как alias_questions / question_scope / owned question survivor slot.

### DUPLICATE
Две surfaces/candidates уже существуют, но отвечают на один intent.

Признаки:
- title различается, но объект ответа тот же;
- canonical_question различается, но expected answer тот же;
- answer одного повторяет answer другого другими словами;
- source evidence совпадает или сильно пересекается.

Действие:
- relation_type = "duplicates" или "near_duplicate";
- recommended_action = "merge_same_surface";
- выбрать survivor по качеству title/question/answer/evidence.

### SHORT_LONG_VARIANT
Одна карточка — краткая формула, другая — расширенная версия того же ответа.

Признаки:
- short_answer одной содержится по смыслу в answer другой;
- отличие только в уровне подробности;
- расширенная версия не отвечает на отдельный follow-up question.

Действие:
- merge_same_surface;
- short_answer survivor = краткая версия answer;
- merged surface не должна остаться отдельной видимой карточкой.

### PARENT_CHILD
Одна surface обзорная, другая отвечает на самостоятельный под-вопрос.

Признаки:
- child можно спросить отдельно;
- child имеет собственный answer value;
- child добавляет отдельный аспект: цена, лимит, канал, интеграция, настройка, процесс, ошибка, роль, ограничение.

Действие:
- relation_type = "umbrella_contains" или "specializes";
- не merge.

### DETAIL_OF
Одна surface детализирует часть другой, но является самостоятельным follow-up answer.

Действие:
- keep separate;
- relation_type = "specializes" или "overlaps";
- reason обязан объяснить отдельный answer value.

### SIBLING_SEPARATE
Одна тема, но разные customer intents.

Признаки:
- разные actions/decisions;
- разные условия;
- разные процессы;
- разные цены/лимиты;
- разные failure modes;
- разные integrations/channels.

Действие:
- keep separate;
- relation_type = "sibling" или "overlaps".

### FULLY_SEPARATE
Разные темы или нет общей decision value.

Действие:
- no relation или relation_type = "unrelated".

## Правило сомнения
Если сомневаешься между alias и new answer slot — выбирай alias.
Если сомневаешься между duplicate и sibling — сравни factual_answer_core:
- один factual answer => duplicate / short-long;
- разные самостоятельные factual answers => parent/child/sibling/separate.

## Обязательные metadata поля
Где output schema позволяет metadata, заполняй:
- customer_intent;
- factual_answer_core;
- alias_questions;
- intent_taxonomy;
- duplicate_of;
- short_long_variant_of;
- keep_separate_reason.

Не выдумывай эти поля, если нет evidence, но если они есть — следующие стадии обязаны их учитывать.

# DISCOVERY_STAGE_STATE_RULES_V1

На discovery stage ты выявляешь initial answer slots в source_unit.

Обязательная процедура:
1. Прочитай INPUT_JSON.compilation_context.known_answer_slots.
2. Для каждого потенциального candidate сначала сформулируй customer_intent.
3. Сформулируй factual_answer_core.
4. Сравни с known_answer_slots и candidates внутри текущего source_unit.
5. Если это SAME_INTENT_ALIAS или SHORT_LONG_VARIANT — не создавай новую карточку.
6. Если нужен candidate, в candidate.metadata запиши customer_intent, factual_answer_core, alias_questions, intent_taxonomy, keep_separate_reason.
7. Новый candidate допустим только для самостоятельного customer intent с отдельным answer value.

Запрещено создавать отдельный candidate только потому что в документе новый heading, другое название, другая длина ответа или другая формулировка вопроса.
