NODE: faq_surface_section_findings

Ты обрабатываешь ОДИН markdown-раздел документа и превращаешь его в findings для регистра клиентских вопросов.

Цель:
понять, какие реальные вопросы клиента этот раздел помогает закрыть, и как эти ответы соотносятся с уже найденными вопросами в registry.

Вход:

source_unit — один markdown-раздел: title, body, raw_text, children, source_refs, source_chunk_indexes, section_path.
registry_snapshot — компактный список уже известных вопросов, кратких ответов, вариантов формулировок, evidence refs и parent/child/duplicate/overlap relations.
match_context — необязательные подсказки предварительного matching.

Работай по шагам.

Шаг 1. Найди answer-bearing spans.
Answer-bearing span — это фраза, абзац, список или блок раздела, который содержит знание, полезное для ответа клиенту.

Шаг 2. Сгруппируй эти spans по customer intent.
Customer intent — это реальный вопрос или намерение клиента, на которое ассистент должен уметь ответить.
Один раздел может:

не дать ни одного нового intent;
дать один intent;
дать несколько intent;
повторить уже известный intent другими словами;
добавить деталь к уже известному intent;
раскрыть более узкий child intent;
дать более общий umbrella / parent intent.

Шаг 3. Сравни каждый intent с registry_snapshot.
Для каждого найденного intent выбери действие:

new — раздел отвечает на новый клиентский вопрос, которого ещё нет в registry.
adds_evidence — раздел не создаёт новый вопрос, но даёт новую цитату, пример, условие или подтверждение для уже известного ответа.
extends_existing — раздел добавляет новую смысловую часть к уже известному вопросу.
refines_existing — раздел уточняет границы, формулировку или условия уже известного вопроса.
duplicate_of — раздел повторяет уже известный вопрос и ответ без существенной новой информации.
child_of — раздел отвечает на более узкий вопрос внутри уже известной umbrella/parent surface.
parent_of — раздел формулирует более общий вопрос, который покрывает уже известную child/specific surface.
umbrella_for — раздел даёт обзорный вопрос/ответ, который группирует несколько известных или новых child intents.
overlaps — тема пересекается с известной surface, но клиентский вопрос отличается; это не merge.

Шаг 4. Отдели surface от служебных ролей.
Surface — это customer-answerable intent: вопрос клиента + ответ, который можно найти через retrieval.
Служебные роли внутри документа не являются surface. Они могут быть источником полезного текста, но не карточкой сами по себе.

Примеры служебных ролей:

factual_answer_core — используй как evidence или answer core.
short_answer / "Короткий ответ клиенту" — используй как short_answer.
customer_intent — используй как canonical_question или variant.
expected_topic / "Ожидаемая тема" — используй как topic hint или variant.
test_question / "Тестовый вопрос" — используй как variant/test question только если это реальный клиентский вопрос.

Шаг 5. Собери finding.
Для каждого finding обязательно укажи:

какой клиентский вопрос закрывается;
новый это вопрос или связь с существующим registry entry;
какой bounded answer можно дать строго по этой секции;
какую новую информацию секция добавляет, если вопрос уже известен;
точные evidence_quotes из source_unit;
source_refs и source_chunk_indexes из source_unit;
почему это именно такой action.

Пример 1 — секция расширяет существующий вопрос.

Registry:

surface_key: product_definition
canonical_question: "Что такое продукт?"
short_answer: "Система превращает документы бизнеса в управляемую AI-базу знаний."

Section:
"Продукт не просто отвечает в чате. Он также показывает владельцу, что AI понял из документов, позволяет исправить карточки до публикации и проверить качество поиска."

Правильный результат:

action: extends_existing
target_surface_key: product_definition
canonical_question: "Что такое продукт?"
answer_delta: "продукт показывает, что AI понял из документов, позволяет исправить карточки до публикации и проверить качество поиска"
variants: ["что делает продукт", "зачем нужна платформа", "как продукт помогает с базой знаний"]
evidence_quotes: точные цитаты из section

Пример 2 — секция создаёт child-вопрос.

Registry:

surface_key: knowledge_curation
canonical_question: "Как управлять AI-базой знаний?"

Section:
"Ручной merge позволяет выбрать несколько похожих карточек, указать что сохранить, что убрать, и собрать один итоговый ответ."

Правильный результат:

action: child_of
target_surface_key: knowledge_curation
local_surface_key: manual_merge
canonical_question: "Как работает ручное слияние карточек?"
surface_kind: "curation"
answer: ответ только про ручной merge
parent_surface_key: knowledge_curation

Пример 3 — секция создаёт umbrella.

Registry:

surface_key: telegram_channel
canonical_question: "Как работает Telegram-ассистент?"
surface_key: web_panel
canonical_question: "Для чего нужна веб-панель?"

Section:
"Платформа объединяет клиентский канал, веб-панель команды и управляемую базу знаний в единый процесс обработки обращений."

Правильный результат:

action: umbrella_for
local_surface_key: support_workflow_overview
canonical_question: "Как устроен общий процесс клиентской поддержки?"
child_surface_keys: ["telegram_channel", "web_panel"]
answer: обзорный ответ, который не заменяет child-карточки

Пример 4 — служебная роль не становится surface.

Section:
"customer_intent: что такое продукт
factual_answer_core: система превращает документы бизнеса в управляемую AI-базу знаний
short_answer: управляемая база знаний для AI-ассистента"

Правильный результат:

одна finding/surface для вопроса "Что такое продукт?"
customer_intent становится canonical_question или variant
factual_answer_core становится answer/evidence
short_answer становится short_answer
НЕ создавать отдельные surfaces с title factual_answer_core, short_answer или customer_intent

Верни только JSON object.

JSON contract:
{
"findings": [
{
"action": "new | extends_existing | refines_existing | adds_evidence | child_of | parent_of | umbrella_for | duplicate_of | overlaps",
"target_surface_key": "existing registry surface key when action points to existing surface, otherwise empty string",
"local_surface_key": "stable local key for new proposed surface, otherwise empty string",
"title": "human readable title for the customer-answerable intent",
"canonical_question": "customer-answerable canonical question",
"surface_kind": "umbrella | child | specific | standalone | definition | procedural | safety | handoff | integration | channel | document_upload | curation | retrieval_quality | service_limits | other",
"answer": "bounded answer based only on this source_unit",
"short_answer": "compact customer-facing answer",
"answer_delta": "new information added to an existing surface, otherwise empty string",
"answer_scope": "what this answer covers",
"question_scope": "what user questions this surface owns",
"exclusion_scope": "nearby questions this surface must not own",
"variants": ["user-like question variants and useful topic hints"],
"parent_surface_key": "existing/proposed parent key if known, otherwise empty string",
"child_surface_keys": ["existing/proposed child keys if known"],
"evidence_quotes": ["exact quotes from source_unit"],
"source_refs": ["source refs from this source_unit only"],
"source_chunk_indexes": [],
"confidence": 0.0,
"reason": "brief evidence-grounded reason",
"warnings": []
}
],
"warnings": [],
"metrics": {
"source_unit_processed": true,
"finding_count": 0,
"new_count": 0,
"existing_update_count": 0,
"duplicate_count": 0,
"role_labels_absorbed": 0
}
}

Sanity rules:

Обрабатывай только переданный source_unit.
Не возвращай весь registry.
Не возвращай весь документ.
Не создавай surface из названия служебного поля.
Если служебное поле содержит полезный текст, используй его как answer, short_answer, variant, topic hint или evidence.
Не возвращай fragments[] или surfaces[] в этом node.
