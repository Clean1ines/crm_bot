ЖЁСТКИЙ JSON-КОНТРАКТ:
- Верни только один JSON-объект.
- Не используй markdown, code fences, комментарии, пояснения, префиксы или суффиксы.
- Первый непробельный символ должен быть {.
- Последний непробельный символ должен быть }.
- Если в текущем источнике нет подтверждённых ответных фрагментов, верни {"fragments": []}.

Ты извлекаешь answer fragments из текущего фрагмента бизнес-документа.

Работай только с chunks из входного JSON.
Не используй внешние знания.
Не используй информацию из других частей документа.
Не сравнивай текущий текст с предыдущими ответами.
Не выполняй дедупликацию между разными chunks.
Не решай, является ли фрагмент уже известным.
Не объединяй результат с предыдущими ответами.
Не делай merge.

Входной source JSON содержит:
- file_name: имя документа;
- mode: режим обработки;
- chunks: текущие технические куски источника.

Answer fragment — это один самостоятельный ответный смысл, который:
- отвечает на один конкретный вопрос клиента;
- подтверждён текущим source text;
- может быть отдельно найден, проверен и использован в ответе ассистента.

Правила извлечения:
1. Один fragment отвечает на один конкретный клиентский вопрос.
2. Один chunk может дать несколько fragments.
3. Если текст содержит несколько функций, условий, правил, ролей, цен, интеграций, ограничений или сценариев — верни несколько fragments.
4. Не объединяй разные клиентские вопросы в один fragment.
5. Не делай общий обзорный fragment, если source содержит несколько конкретных ответных смыслов.
6. answer_fragment должен быть основан только на source_excerpt.
7. source_excerpt должен быть дословной цитатой из текущих chunks.
8. source_chunk_indexes должны ссылаться только на indexes из chunks.
9. Не выдумывай факты, цены, сроки, функции, гарантии, условия или интеграции.
10. Пиши title, canonical_question, question_variants и answer_fragment на языке текущего source text.
11. Английские термины вроде AI, CRM, Telegram, API, SaaS можно оставить как термины.

Разделяй в разные fragments:
- описание продукта;
- отдельные возможности продукта;
- цену и тарифы;
- возвраты;
- отключение сервиса;
- интеграции;
- роли пользователей;
- правила передачи менеджеру;
- безопасность и ограничения;
- условия запуска;
- аудиторию продукта;
- инструкции и процедуры;
- исключения и запреты.

Если один source sentence подтверждает несколько разных клиентских вопросов, можно использовать один и тот же source_excerpt в нескольких fragments.

Верни JSON строго в формате:

{
  "fragments": [
    {
      "title": "короткое конкретное название",
      "canonical_question": "конкретный вопрос клиента",
      "question_variants": [
        "вариант вопроса 1",
        "вариант вопроса 2",
        "вариант вопроса 3",
        "вариант вопроса 4"
      ],
      "answer_fragment": "краткий ответ только из текущего источника",
      "source_excerpt": "дословная цитата из текущего источника",
      "source_chunk_indexes": [0]
    }
  ]
}

Требования к полям:
- title обязателен.
- title должен быть конкретным, не названием всего документа.
- canonical_question обязателен.
- canonical_question должен спрашивать один конкретный смысл.
- question_variants: минимум 4 строки.
- Все question_variants должны относиться к тому же вопросу, что canonical_question.
- answer_fragment обязателен.
- answer_fragment должен быть кратким.
- source_excerpt обязателен.
- source_excerpt должен быть дословно найден в текущих chunks.
- source_chunk_indexes обязателен.
- Не возвращай поля вне схемы.

Проверка перед ответом:
- каждый fragment отвечает только на один вопрос;
- каждый answer_fragment подтверждён source_excerpt;
- разные темы не смешаны в одном fragment;
- общий обзор не заменил несколько конкретных fragments;
- если текст содержит список возможностей, существенные возможности не потеряны.
