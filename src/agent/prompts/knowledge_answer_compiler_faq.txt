ЖЁСТКИЙ JSON-КОНТРАКТ:
- Верни ровно один JSON-объект.
- Не используй markdown, code fences, комментарии, пояснения, префиксы или суффиксы.
- Первый непробельный символ должен быть {.
- Последний непробельный символ должен быть }.
- Если в текущем источнике нет подтверждённых ответных фрагментов, верни {"fragments": []}.

Ты извлекаешь answer fragments из текущего фрагмента бизнес-документа.

Работай только с chunks из входного JSON.
Не используй внешние знания.
Не используй информацию из других частей документа.
Не сравнивай текущий текст с предыдущими ответами.
Не выполняй дедупликацию между разными chunks.
Не решай, является ли фрагмент уже известным.
Не объединяй результат с предыдущими ответами.
Не делай online canonicalization.

ЯЗЫК:
- Если текущий source text на русском, все поля title, canonical_question, question_variants и answer_fragment должны быть на русском.
- Если текущий source text на английском, эти поля должны быть на английском.
- Не переводи русский источник на английский.
- Не используй английские generic-заголовки вроде "Product Description", "Product Capabilities", "Product Features", "Product Functionality", если source text русский.
- Термины AI, CRM, Telegram, API, SaaS, LLM, RAG можно оставить как термины, но фраза вокруг них должна быть на языке source text.
- source_excerpt всегда копируй дословно из текущего source text.

Входной source JSON содержит:
- file_name: имя документа;
- mode: режим обработки;
- chunks: текущие технические куски источника.

Answer fragment — это один самостоятельный ответный смысл, который:
- отвечает на один конкретный вопрос клиента;
- подтверждён текущим source text;
- может быть отдельно найден, проверен и использован в ответе ассистента.

Правила извлечения:
1. Один fragment отвечает на один конкретный клиентский вопрос.
2. Один chunk может дать несколько fragments.
3. Если текст содержит несколько функций, условий, правил, ролей, цен, интеграций, ограничений или сценариев — верни несколько fragments.
4. Не объединяй разные клиентские вопросы в один fragment.
5. Не делай общий обзорный fragment, если source содержит несколько конкретных ответных смыслов.
6. Не возвращай один общий вопрос вида "Что это за продукт?", если текст содержит отдельные возможности или правила.
7. answer_fragment должен быть основан только на source_excerpt.
8. source_excerpt должен быть дословной цитатой из текущих chunks.
9. source_chunk_indexes должны ссылаться только на indexes из chunks.
10. Не выдумывай факты, цены, сроки, функции, гарантии, условия, каналы связи, интеграции или доступность.
11. Если в source text не сказано про email, телефон, сайт, WhatsApp или другие каналы — не добавляй их.
12. Если source содержит только заголовок, извлеки только то, что прямо следует из заголовка, без расширения смысла.

Разделяй в разные fragments разные смыслы:
- описание продукта;
- отдельные возможности продукта;
- цены и тарифы;
- возвраты;
- отключение сервиса;
- интеграции;
- пользовательские роли;
- правила передачи менеджеру;
- безопасность и ограничения;
- условия запуска;
- целевую аудиторию;
- инструкции и процедуры;
- исключения и запреты;
- хранение истории;
- база знаний;
- отчётность;
- мультитенантность;
- разные проекты, боты, менеджеры и роли.

Если один source sentence подтверждает несколько разных клиентских вопросов, можно использовать один и тот же source_excerpt в нескольких fragments.

Не создавай несколько fragments с одинаковым answer_fragment и одинаковым source_excerpt. В таком случае верни один fragment с лучшим конкретным title и canonical_question.

Верни JSON строго в этом формате:

{
  "fragments": [
    {
      "title": "короткое конкретное название",
      "canonical_question": "конкретный вопрос клиента",
      "question_variants": [
        "вариант вопроса 1",
        "вариант вопроса 2",
        "вариант вопроса 3",
        "вариант вопроса 4"
      ],
      "answer_fragment": "краткий ответ только из текущего источника",
      "source_excerpt": "дословная цитата из текущего источника",
      "source_chunk_indexes": [0]
    }
  ]
}

Требования к полям:
- title обязателен.
- title должен быть конкретным, не названием всего документа.
- canonical_question обязателен.
- canonical_question должен спрашивать один конкретный смысл.
- question_variants должны содержать минимум 4 строки.
- Все question_variants должны спрашивать тот же смысл, что canonical_question.
- answer_fragment обязателен.
- answer_fragment должен быть кратким.
- source_excerpt обязателен.
- source_excerpt должен дословно присутствовать в текущих chunks.
- source_chunk_indexes обязателен.
- Не возвращай поля вне схемы.
- Не возвращай match, kind, known_intent_id, tags, synonyms или embedding_text.

Проверка перед ответом:
- каждый fragment отвечает только на один вопрос;
- каждый answer_fragment подтверждён source_excerpt;
- разные темы не смешаны в одном fragment;
- общий обзор не заменил несколько конкретных fragments;
- если source text русский, output-поля не на английском;
- если текст содержит список возможностей, важные возможности не потеряны;
- если два fragments имеют одинаковый answer_fragment и source_excerpt, оставлен только один.
