ЖЁСТКИЙ JSON-КОНТРАКТ:
- Верни ровно один JSON-объект.
- Не используй markdown, code fences, комментарии, пояснения, префиксы или суффиксы.
- Первый непробельный символ должен быть {.
- Последний непробельный символ должен быть }.
- Если в текущем source unit нет подтверждённых ответных фрагментов, верни {"fragments": []}.

Ты извлекаешь grounded answer fragments из текущего source unit документа.

Работай только с chunks из входного JSON.
Не используй внешние знания.
Не используй информацию из других частей документа.
Не сравнивай текущий текст с предыдущими ответами.\nНе сравнивай текущий source unit с предыдущими ответами.
Не выполняй межсекционную дедупликацию.
Не объединяй результат с предыдущими ответами.
Не делай canonical publication.
Не выдумывай факты, цены, сроки, функции, гарантии, условия, каналы связи, интеграции или доступность.

Входной source JSON содержит:
- file_name: имя документа;
- mode: режим обработки;
- chunks: один или несколько source units.

Chunk может быть:
- обычным техническим текстовым фрагментом;
- структурированным Markdown source unit с section_title, section_body, children, section_path, source_excerpt, source_refs.

Answer fragment — это полный самодостаточный ответный фрагмент, который:
- отвечает на один конкретный user intent;
- полностью подтверждён текущим source unit;
- может быть отдельно сохранён как raw answer candidate;
- может использовать весь source unit как evidence, если весь source unit нужен для полного ответа;
- может иметь source_excerpt, полностью совпадающий с answer_fragment;
- может иметь source_excerpt, полностью совпадающий со всем source unit.

Один chunk может дать несколько fragments.\nОдин source unit может дать:
- ноль fragments, если в нём нет самостоятельного ответа;
- один fragment, если весь source unit отвечает на один intent;
- несколько fragments, если один source unit содержит несколько самостоятельных grounded intents.

Если один и тот же source_excerpt подтверждает несколько разных user intents, можно использовать один и тот же source_excerpt в нескольких fragments.

ЯЗЫК:
- Если текущий source text на русском, все поля title, canonical_question, question_variants и answer_fragment должны быть на русском.
- Если текущий source text на английском, эти поля должны быть на английском.
- Не переводи русский источник на английский.
- Не используй английские generic-заголовки вроде "Product Description", "Product Capabilities", "Product Features", если source text русский.
- Термины AI, CRM, Telegram, API, SaaS, LLM, RAG можно оставить как термины, но фраза вокруг них должна быть на языке source text.
- source_excerpt всегда копируй дословно из текущего source text.

ОБЩИЕ ПРАВИЛА ИЗВЛЕЧЕНИЯ:
1. Один fragment отвечает на один конкретный клиентский вопрос или один конкретный user intent.
2. answer_fragment должен быть полным ответом на canonical_question, а не кратким обрывком.
3. answer_fragment может быть длинным, если для полного ответа нужны правила, условия, списки, примеры или ограничения.
4. source_excerpt должен дословно подтверждать answer_fragment.
5. Не создавай fragment, где answer_fragment не отвечает на canonical_question.
6. Не создавай fragment из вопроса, заголовка или пункта списка, если в текущем source unit нет ответа на этот intent.
7. Не превращай пример в отдельную карточку, если пример только иллюстрирует родительское правило.
8. Не превращай проверочный вопрос в production answer, если он является частью тестового набора.
9. Если секция сама является знанием о тестировании, правилах, проверке, RAG-поиске, поведении системы или структуре базы знаний — можно создать fragment об этой секции как об отдельном знании.
10. Если source unit содержит несколько самостоятельных Q/A блоков, извлеки несколько fragments.
11. Если source unit содержит одно правило с примерами, извлеки один fragment про правило, а примеры включи в answer_fragment.
12. Если source unit содержит тестовый набор, извлеки один fragment про то, как использовать этот тестовый набор, а вложенные вопросы оставь внутри answer_fragment/source_excerpt.
13. Если source unit содержит FAQ container с самостоятельными child Q/A, извлеки fragments по этим child Q/A.
14. Не создавай общий обзорный fragment, если source unit явно содержит несколько самостоятельных ответов.
15. Не дроби один цельный ответ на несколько fragments только потому, что внутри есть список.

MARKDOWN-СТРУКТУРА:
Если chunk содержит section_title, section_body и children:
- Сначала определи роль текущей секции по структуре и содержанию.
- section_title — это контекст source unit.
- section_body — текст родительской секции.
- children — вложенные подразделы, обычно Markdown ###.

Допустимые section_shape:
- faq_container;
- single_answer;
- test_suite;
- instruction_reference;
- mixed;
- unknown.

section_shape нужен только для выбора стратегии извлечения. Не возвращай section_shape вне metrics.

FAQ CONTAINER:
- Если children являются самостоятельными Q/A блоками, создай отдельный fragment для каждого child.
- canonical_question обычно соответствует смыслу child title.
- answer_fragment должен полностью отвечать на canonical_question, используя child body.
- source_excerpt обычно равен полному child subsection.
- parent section_title сохраняй как контекст, но не превращай сам по себе в fragment, если parent body не содержит отдельного ответа.

SINGLE ANSWER / RULE / REFERENCE:
- Если вся секция отвечает на один intent, создай один fragment.
- answer_fragment должен включать всё, что нужно для полного ответа: правило, условия, списки, примеры, запреты, исключения.
- source_excerpt может быть всей секцией.
- Не создавай отдельные fragments из пунктов списка, если они только примеры или части одного правила.

TEST SUITE:
- Если секция содержит тестовые вопросы, группы вопросов, expected topics, preview/eval checks или проверочные сценарии, не превращай вложенные вопросы в отдельные customer FAQ answers.
- Создай fragment о самом тестовом наборе, если source unit объясняет, как он используется или что проверяет.
- canonical_question должен спрашивать про тестирование/проверку базы знаний/preview/eval, а не повторять отдельный проверочный вопрос из списка.
- answer_fragment должен содержать или суммировать группы вопросов, ожидаемые темы и назначение теста.
- source_excerpt может быть всей test-suite секцией.

MIXED:
- Если parent body содержит самостоятельное знание, можно создать parent fragment.
- Child fragments создавай только там, где child содержит самостоятельный topic/question и полный grounded answer.
- Не создавай fragments из headings alone.
- Не создавай fragments из examples, если они не являются самостоятельными ответами.

Верни JSON строго в этом формате:

{
  "fragments": [
    {
      "title": "короткое конкретное название",
      "canonical_question": "конкретный вопрос или user intent",
      "question_variants": [
        "вариант вопроса 1",
        "вариант вопроса 2",
        "вариант вопроса 3",
        "вариант вопроса 4"
      ],
      "answer_fragment": "полный самодостаточный ответ, grounded только в текущем source unit",
      "source_excerpt": "дословная цитата из текущего source unit, достаточная для проверки answer_fragment",
      "source_chunk_indexes": [0]
    }
  ],
  "metrics": {
    "section_shape": "faq_container | single_answer | test_suite | instruction_reference | mixed | unknown"
  }
}

Требования к полям:
- title обязателен.
- title должен быть конкретным, не названием всего документа, кроме случаев когда вся секция является одним знанием.
- canonical_question обязателен.
- canonical_question должен соответствовать answer_fragment.
- question_variants должны содержать минимум 4 строки.
- Все question_variants должны спрашивать тот же intent, что canonical_question.
- answer_fragment обязателен.
- answer_fragment должен быть полным ответом на canonical_question.
- answer_fragment может полностью совпадать с source_excerpt.
- source_excerpt обязателен.
- source_excerpt должен дословно присутствовать в текущем source text.
- source_excerpt может быть всей секцией, если вся секция нужна как evidence.
- source_chunk_indexes обязателен.
- source_chunk_indexes должны ссылаться только на indexes из chunks.
- Не возвращай поля вне схемы.
- Не возвращай match, kind, known_intent_id, tags, synonyms или embedding_text.

Проверка перед ответом:
- каждый fragment отвечает на один intent;
- каждый answer_fragment полностью отвечает на canonical_question;
- каждый answer_fragment подтверждён source_excerpt;
- разные самостоятельные intents не смешаны в одном fragment;
- один цельный rule/test/reference answer не раздроблен на примеры;
- FAQ children извлечены отдельно только если child содержит самостоятельный ответ;
- test-suite questions не превращены в customer FAQ answers;
- если source text русский, output-поля не на английском.
