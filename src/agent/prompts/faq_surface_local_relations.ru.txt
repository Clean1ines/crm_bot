Ты строишь локальные отношения между answer slots, найденными в одном source_unit.

Вход:
- source_unit;
- surface_candidates[]. Каждый candidate — это answer slot: один пользовательский интент и один будущий канонический ответ.

Задача:
Определить отношения между candidates по пользовательскому интенту, а не по общей теме.

Типы отношений:
- duplicates: оба candidates отвечают на один и тот же вопрос клиента. Одна версия может быть краткой, другая более полной и с деталями.
- near_duplicate: почти один интент, но есть сомнение.
- umbrella_contains: широкий overview slot покрывает область более узкого slot, а узкий slot отвечает на конкретный вопрос с деталями.
- specializes: source_surface_key является более узкой конкретизацией target_surface_key.
- sibling: одна область или общий overview, но разные вопросы клиента.
- overlaps: пересекаются фактами, но не один answer slot.
- contradicts: факты или условия конфликтуют.
- split_needed: candidate смешивает несколько answer slots.
- unrelated: полезной связи нет.

Правила:
- duplicate ставь только если один и тот же реалистичный вопрос клиента мог бы получить оба ответа.
- Если один ответ является краткой версией другого ответа для того же вопроса, это duplicates.
- Если широкий обзор содержит тему, а узкий candidate раскрывает конкретный вопрос, это umbrella_contains или specializes, не duplicate.
- Широкий overview не должен владеть вопросами, на которые лучше отвечает узкий slot.
- Same domain не означает duplicate.
- overlap не означает merge.

Верни JSON:
{
  "relations": [
    {
      "source_surface_key": "...",
      "target_surface_key": "...",
      "relation_type": "umbrella_contains|specializes|sibling|overlaps|duplicates|near_duplicate|contradicts|unrelated|split_needed",
      "direction": "source_to_target|target_to_source|undirected",
      "confidence": 0.0,
      "reason": "...",
      "source_refs": []
    }
  ],
  "warnings": []
}
