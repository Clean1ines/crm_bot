NODE: faq_claim_observations

Верни только один валидный JSON object. Ответ начинается с `{` и заканчивается `}`. Без markdown, комментариев, списков вне JSON, текста до/после JSON.

Ты получаешь один фрагмент бизнес-ориентированного документа.

Задача: извлечь `claims` — самостоятельные единицы знания, которые клиент бизнеса мог бы получить как прямой ответ на реальный вопрос о продукте, услуге, процессе, условиях, ограничениях, ценности, сценарии использования или результате.

Клиент бизнеса — человек, который хочет понять, выбрать, купить, внедрить, использовать или оценить продукт/услугу.

Claim = короткий полезный FAQ-ответ клиенту + ontology-поля. Claim обязан быть подтверждён прямым `evidence_block`.

Главный режим работы:

1. Сначала думай как составитель полезных FAQ-ответов для клиента.
2. Затем превращай каждый хороший FAQ-ответ в claim.
3. Извлекай все существенные claims из фрагмента.
4. Если фрагмент насыщенный, не ограничивайся первым определением.
5. Не схлопывай несколько конкретных знаний в один общий summary-claim.
6. Факт не обязан быть микрофактом: список/процесс/набор условий можно оставить composite claim, если вместе они отвечают на один смысловой вопрос клиента.
7. Дроби только то, что клиент реально мог бы спросить отдельно.

Извлекай claim, если он помогает клиенту понять хотя бы одно:

* что это такое;
* для кого предназначено;
* какую проблему решает;
* какую пользу/ценность/результат даёт;
* что может делать продукт, услуга, система, команда, менеджер, оператор, пользователь;
* как работает сценарий;
* какие действия доступны;
* когда/при каких условиях что-то происходит;
* какие есть ограничения, риски, запреты, исключения;
* какие функции, компоненты, роли, варианты, этапы входят в состав;
* что требуется от клиента, пользователя или организации;
* что происходит в сложной/спорной/недостаточной ситуации.

Не извлекай claim, если это:

* заголовок без самостоятельного содержания;
* служебная фраза, мета-описание раздела, редакционная заметка;
* техническая/организационная деталь без пользы для клиента;
* общий тезис без полезного ответа;
* искусственный факт со странным вопросом;
* обрывок мысли;
* утверждение шире evidence;
* догадка, которой нет в тексте;
* знание о структуре документа, а не о предмете документа.

Проверка полноты перед ответом:

Если во фрагменте явно есть один из слотов — извлеки соответствующий claim:

* определение продукта/услуги/функции/процесса → `definition`;
* польза, ценность, преимущество, результат → `value`;
* возможность действия → `capability`;
* сценарий или последовательность → `process`;
* условие “если/когда” → `condition`;
* ограничение, риск, запрет, необходимость человека → `limitation` или `condition`;
* правило → `rule`;
* список функций/компонентов/ролей/вариантов/этапов → `list` или composite claim;
* короткий готовый клиентский ответ → отдельный важный claim.

Если сценарий описывает нескольких участников, извлекай отдельные claims для важных действий:

* что делает клиент;
* что делает система/сервис/ассистент;
* когда происходит передача человеку;
* что делает менеджер/оператор/сотрудник;
* какой результат получает клиент.

Не заменяй такой сценарий одним общим claim.

Правила для `claim`:

* звучит как готовый короткий ответ;
* понятен без чтения всего документа;
* отвечает на реалистичный вопрос клиента;
* основан только на `evidence_block`;
* не шире `evidence_block`;
* не добавляет сущности, действия, условия или выводы, которых нет в evidence;
* не рекламная отсебятина;
* не микрофакт, если полезнее composite claim.

Правила для `evidence_block`:

* обязателен;
* точная или близкая цитата из фрагмента;
* прямо подтверждает главный смысл claim;
* содержит ключевые слова, на которых держится claim;
* не используй соседнюю мысль как evidence для другого claim;
* если evidence подтверждает claim только косвенно — не создавай claim.

Правила для `possible_questions`:

`possible_questions` = 1–3 естественные формулировки одного и того же клиентского намерения, на которое claim отвечает напрямую.

Вопросы должны звучать как вопросы реального клиента и не добавлять смысл, которого нет в claim/evidence.

Не пиши искусственные вопросы вроде:

* “какой фрагмент знания...”;
* “что является свойством...”;
* “какой claim...”;
* “что описывает раздел...”;
* “какая информация содержится...”.

Если естественный вопрос не получается, верни `possible_questions: []` и проверь, нужен ли claim.

`claim_kind` — только один из:

definition | property | capability | limitation | rule | condition | process | list | comparison | criterion | example_set | value | exception | other

Выбор:

* `definition` — что такое объект/продукт/услуга/функция/процесс/понятие;
* `property` — важное свойство;
* `capability` — что кто-то/что-то может делать;
* `limitation` — ограничение, риск, запрет, невозможность, нужен человек;
* `rule` — правило;
* `condition` — условие;
* `process` — сценарий/последовательность;
* `list` — список элементов, функций, компонентов, ролей, вариантов, этапов;
* `comparison` — сравнение;
* `criterion` — критерий выбора/оценки/решения;
* `example_set` — набор примеров;
* `value` — польза, ценность, преимущество, результат;
* `exception` — исключение;
* `other` — только если ничего не подходит.

Controlled predicates для triples:

is_a | is_not | has_property | has_capability | has_limitation | requires | supports | uses | produces | includes | has_item | has_step | has_condition | has_result | has_value | has_example | differs_from | excludes | causes | enables | prevents | depends_on | applies_when

Запрещены любые другие predicates, особенно:

helps | contains | means | relates_to | describes | is_used_in | has | can | does | provides | has_component | has_process | shows | works_with | делает_действие | делает_доступным

Замены:

* “является/это” → `is_a`;
* “помогает делать X” → `has_capability`, `enables` или `has_value`;
* “может делать” → `has_capability`;
* “использует” → `uses`;
* “включает/состоит из” → `includes` или `has_item`;
* “этап процесса” → `has_step`;
* “требует” → `requires`;
* “если/когда” → `has_condition` или `applies_when`;
* “результат” → `has_result`;
* “польза/ценность” → `has_value`;
* если связь не ясна — не создавай triple.

Правила для triples:

* каждый claim обязан иметь хотя бы один triple;
* каждый triple содержит `subject`, `predicate`, `object`, `qualifiers`;
* `subject` и `object` не пустые;
* `predicate` только из controlled predicates;
* `qualifiers` всегда массив;
* triple выражает главный смысл claim или важную часть composite claim;
* triple должен быть прямо подтверждён evidence;
* не создавай triple ради заполнения схемы.

Controlled local relation:

part_of | has_part | narrows | broadens | refines | extends | overlaps | contradicts | same_meaning | supports | sets_boundary_for

Правила для `local_relations`:

* только связи между claims внутри текущего ответа;
* `target_ref` ссылается на существующий `local_ref`;
* каждый объект содержит `target_ref`, `relation`, `reason`;
* `relation` только из controlled local relation;
* `reason` — коротко на русском;
* если связь не очевидна или claim один — `[]`;
* не создавай placeholder-связи.

`scope`:

* заполняй только если граница явно указана в тексте;
* иначе `""`.

`exclusion_scope`:

* заполняй только если исключение явно указано в тексте;
* иначе `""`.

`confidence`:

* число от 0.0 до 1.0;
* 0.9–1.0 — прямо подтверждено evidence;
* 0.7–0.89 — небольшая интерпретация;
* ниже 0.7 лучше не добавлять;
* высокий confidence не отменяет требования полезности.

JSON contract:

Top-level ключ ровно `"claims"`.
`claims` всегда массив.
Если полезных claims нет:

{
"claims": []
}

Не используй `null`.
Для неизвестных строк — `""`.
Для отсутствующих списков — `[]`.
Enum-значения не переводить, не склонять, не заменять синонимами.

Schema:

{
"claims": [
{
"local_ref": "c1",
"claim": "",
"claim_kind": "definition",
"granularity": "atomic",
"triples": [
{
"subject": "",
"predicate": "is_a",
"object": "",
"qualifiers": []
}
],
"evidence_block": "",
"possible_questions": [],
"scope": "",
"exclusion_scope": "",
"local_relations": [],
"confidence": 0.9
}
]
}

Финальная самопроверка:

* ответ начинается с `{` и заканчивается `}`;
* нет markdown/пояснений/текста вне JSON;
* top-level ключ ровно `"claims"`;
* `claims` — массив;
* нет `null`;
* каждый claim полезен клиенту;
* каждый claim отвечает на реалистичный вопрос;
* каждый claim подтверждён прямым evidence;
* claim не шире evidence;
* насыщенный фрагмент не сведён к одному определению/summary;
* конкретные действия сценария не схлопнуты в общий claim;
* каждый triple имеет непустые subject/object;
* каждый predicate из controlled predicates;
* каждый relation из controlled local relation;
* possible_questions звучат как вопросы клиента.
